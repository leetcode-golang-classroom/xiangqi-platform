## Context

規則引擎是平台所有功能的地基（單機、復盤、AI、線上）。它必須純邏輯、零 UI/網路依賴、可被多端與未來他語言實作共用。完整元件設計（職責、介面、循序圖）見 `docs/DESIGN.md`。

## Goals / Non-Goals

**Goals**
- 正確且可驗證的象棋規則：走法產生、合法性、勝負判定。
- 語言中立契約：座標、FEN、UCCI、棋譜格式、`RuleEngine` 介面，可跨語言移植。
- 不可變狀態，利於復盤與 AI 搜尋。

**Non-Goals**
- UI/渲染（屬 `client`）、AI 搜尋演算法（屬 `player`）、線上傳輸（屬 `server`）。
- 棋力強度與評估函數（後續階段）。

## Decisions

- **盤面表示：一維陣列 `[90]Piece`（rank*9+file）**，而非二維或 bitboard。理由：簡單、可移植、易由 FEN 對映；象棋分支量不大，效能足夠。bitboard 留待 AI 階段如有需要再優化。
- **棋子以 FEN 字母（byte）表示**：與 FEN/序列化天然對齊，跨語言一致。
- **`ApplyMove` 不可變、回傳新 `Game`**：復盤與 AI 樹搜尋可安全共享歷史節點；代價是每步複製盤面（90 bytes，可接受）。
- **走法產生分兩層**：先產生「擬合法」走法（僅看走子規則與蹩馬腿/塞象眼/炮架），再過濾「走後自將/飛將」。理由：邏輯清晰、與規格的 Requirement 對應。
- **將死/困斃統一以「無合法走法」判定**，再依是否被將軍區分結果，避免重複邏輯。
- **記譜雙軌**：儲存用 UCCI（機器標準、與未來 UCCI 引擎相容），顯示用中文（即時由座標轉換）。
- **一致性以 `conformance/*.json` 黃金案例驅動**：Go 與未來他語言實作共用同一份案例，table-driven harness 載入斷言。

## Risks / Trade-offs

- [長將/和棋規則的細節（重複局面判定、自然限著上限）易有歧義] → 先以可驗證子集（連續將軍計數、步數上限）落地，棋規細節以增量 Requirement + fixture 補齊。
- [中文記譜在多子同列（前/後、進/退/平）有複雜命名規則] → 先支援基本情形，複雜消歧另立 fixture 漸進完善。
- [不可變複製的效能] → 對單機/復盤無虞；AI 階段若成瓶頸再導入 make/unmake 或 bitboard。

## Migration Plan

純新增，無既有行為變更。實作順序：FEN/UCCI 轉換 → 各子走法產生 → 自將過濾 → 結果判定 → 記譜/復盤 → 中文記譜。每步讓對應的 conformance 測試由紅轉綠。
